Scope of This Document

This guide is intended for day-to-day use of the TOS console. For information on running and administering a TOS system, see the Administrator’s Guide . For information on developing applications to run under TOS, see the Developer’s Guide. For a technical overview of TOS design, implementation, internal structure, and possible extensions, see Tyrone’s Operating System: An Educational Operating System in Java.

Introduction to TOS

Tyrone’s Operating System, TOS, was prepared as a thesis for COSC 4F90 at Brock University. It is intended to be used in senior-year undergraduate courses in operating system implementation.

TOS is a distributed operating system. Unlike conventional operating systems, which run on only one computer, TOS can run on any number of different computers simultaneously, as long as they are connected over a network running the TCP/IP protocol suite. All the computers running TOS will share a common file system and common access to the same system objects. Programs running on one computer can coordinate their activities and send information to programs on another quickly and easily.

Another difference between most operating systems is that TOS is not a native operating system. It is not booted by a computer when it is turned on and it does not deal directly with the computer’s hardware. Instead, it runs on top of the computer’s native operating system.

The basic modules enabling you to use TOS have been prepared for the UNIX family, Microsoft Windows 95/NT, and the Macintosh System 7.6 or later.

Connecting to TOS

Note: Currently the TOS console is not supported on the Macintosh. See the Final Report for an explanation of the technical reasons for this.

JAVA Environment Variable

The JAVA environment variable must be set to the name of the executable containing the console version of the Java run-time engine. If you are using the Sun JDK or JRE, you may use either "java" or "jre" for this variable. If you are using the Microsoft VM for Java, use "jview", located in your Windows directory. Do not use the windowed versions of the run-time engine, such as "javaw" or "wjview".

CLASSPATH Environment Variable

Your CLASSPATH environment variable must also point to the same directory containing the TOS executables. This directory will contain a subdirectory called tos which contains the Java class files that implement TOS. Make sure that your CLASSPATH points to the parent of tos, not to tos itself.

Starting the Console

To start the TOS console, simply type Console at any command-line prompt. You may also make use of any other method your host offers for starting applications, such as creating a desktop shortcut in Windows or an alias in UNIX.

You will see a command-line prompt saying simply TOS:>. This is the TOS prompt. From here, you can enter commands simply by typing the command name, its parameters, and <enter> or <return>.

Connecting a Console to a Launcher

You will not be able enter any commands in TOS (except exit) without connecting to a launcher. The syntax for doing this is:

connect host[:port]

where host refers to either the qualified network name or IP address or the machine running a TOS launcher, and port refers to the TCP/IP port number the launcher is running on. If you do not specify a port number, it will default to 1099.

So if you are connecting to a launcher running on port 1101 of a host with IP address 139.57.96.16, you would enter

connect spartan.ac.brocku.ca:1101

If you were running on a machine called sandcastle and wished to connect to a launcher also running on sandcastle, on port 1105, enter

connect sandcastle:1105

Dynamic TCP/IP Addressing

The machine running the launcher you are connecting to must be linked to the one you are running the console from on a network supporting TCP/IP.

If that machine uses dynamic TCP/IP addressing, which is generally the case for machines using a dial-up network connecting, and often also for PC or Macintosh workstations, you must make sure that the IP address you are using corresponds to the one in use at the time. This may preclude using the connect command in a script file.

Reconnection

If you wish during a single console session to connect to a different launcher, or even if you need to reconnect to the first because it crashed and was restarted, you must use the disconnect command before running connect again.

This is because each connection to a launcher does not persist across different execution instances of the launcher. If the launcher stops, you must disconnect and connect again.

The TOS File System

Single Global Namespace

TOS’ file system is based largely on the UNIX model. There can be any number disk servers on different machines within the TOS system, but system administrators may mount different disks to any directory path (see the Administrator’s Guide for further details). The overall effect is to resemble a single large virtual disk.

This is in contrast to Microsoft and Apple operating systems, where every path name includes a reference to the physical disk partition on which a file is located.

Directories

As in UNIX, there is a central root directory referred to as /. From there, a hierarchical tree structure of directories is laid out, using / as the path separator. There is no fixed limit on the number of files within any single directory, other than the limits applicable to the physical disk the file or directory is actually located on (see below).

File Names

You can refer to a file by either its absolute path name or its relative path name, the name comparable to your working directory. This follows the same conventions as in both UNIX and MS-DOS. So a file named foo located in directory mydir has the absolute path name /mydir/foo. If you are in the directory otherdir, another subdirectory of root, you may use the .. operator to refer to your immediate parent. In that case foo’s path, relative to you, becomes ../mydir/foo. A file myfile located in otherdir can be simply referred to as myfile (relative) or /otherdir/myfile.

Files may have a maximum of 255 characters in their names. TOS will not stop you from creating two files in the same directory with the same name, but if you do you will not be able to access the second file until you move or rename the first.

File names may not contain either the path separator character (/) or the colon (:) character, which TOS uses for internal operations. Spaces also will not be recognized as part of a filename and will result in a syntax error. You are also advised not to use the path separator character for your host (\ on Windows).

Physical Disk Servers

You may often find that files are unavailable. The most common reason for this is that the TOS disk server in which the file’s directory is mounted is no longer running. You will not be able to access the file until it is started again. See the Administrator’s Guide for instructions.

Each of the disks has a limit on both the overall quantity of data it can hold and the number of files it can hold. These restrictions serve as the only limits on the size of any individual file or directory – neither can grow so large as to break their disk’s limits. System administrators set these limits when the disk is created (see the Administrator’s Guide ). They can view this information later but not change it.

Types of Files

Files can contain binary or character data. Character files use the Unicode character encoding system, which supports a wide variety of human languages, each with its range of symbols and accents.

Security in the File System

Entering Passwords

Security in TOS is based on assigning passwords to files. Any user can issue a password on a file. The command to do this is:

protect filename [oldpassword]

You must enter the correct value for oldpassword if the file already has a password. If it does not, omit this parameter.

You will then be prompted for a new password. The new password may be of any length and may use any combination of characters except spaces. The password characters are not hidden on the screen, so be sure your enemies aren’t watching.

Removing Passwords

To remove a password, simply use protect and leave the new password blank when prompted to enter one.

File Permissions

Once you have placed a password on a file, you may select the file’s permissions. These permissions place limits on what users can do with the file without entering its password.

Read permission enables users to be able to read the contents of the file. They can access the file’s contents but may not change its contents.

Write permission enables users to change the contents of the file. This includes being able to completely overwrite its contents if they wish.

Execute permission is relevant only to Java class files and TOS script files. It enables the class files or scripts to be run from the TOS console.

Setting Permissions

Merely choosing a password does not, by itself, have an effect on the ability of other users to access a file without knowing the password. You must set the file’s permission bits, which again follow the model of UNIX. The syntax is:

chmod filename permissions password

The permissions parameter is a logical OR of three bits. The left-most bit represents read permission, the middle bit is write permission, and the right-most bit is for execute permission.

For the uninitiated in bitwise arithmetic, here are the values to be used for difference combinations:

  • 0 – None
  • 1 – Execute only
  • 2 – Write only
  • 3 – Write and execute
  • 4 – Read only
  • 5 – Read and execute
  • 6 – Read and write
  • 7 – Read, write, and execute

Values 8 and above will produce an error message.

Accessing a Protected File

If the file’s permissions permit the operation you wish to execute, you do not need to enter a password when accessing the file. If they do, the last parameter in the command you enter must be the password.

Consider trying to use the export command on a file named myfile, with password mypassword, that has write-only permission. Export requires read permission, which the file doesn’t have. So the user must supply its password, as follows:

export myfile hostfile mypassword

(See the section Importing and Exporting Files for details on export.)

Note: TOS does not yet make use of passwords for operations involving more than one file, that is copy and move. Files being copied and moved must be left unprotected in the current version of TOS.

Navigating in TOS

Working Directory

Your current working directory will be contained within the TOS prompt. It will be prepended to any non-absolute pathname you enter for any command.

Moving Around

Change the working directory using the cd command. TOS will automatically locate the correct TOS disk server and connect you to it behind the scenes if it is available. If it is not, you will not be able to complete the cd operation and a warning message will appear. As always, you may use either absolute or relative names to move around.

Basic File Operations

Copy, or cp, creates a second copy of a file at another location.

Move, or mv, creates a copy of a file at another location and deletes the original file.

Delete, or del, or rm, deletes a file. Write permission is required.

Directory Operations

Mkdir, or md, creates a directory.

Rmdir, or rd, removes a directory.  The directory must already be empty.

Editing Files in TOS

You can use the edit command to invoke the TOS editor. This is a GUI program, so it may only be used if you have access to a windowing environment, which rules out running it from a telnet session.

The author of TOS, curse him, intended the editor only for quick edits. As a result, it has deliberately been left primitive, with not even basic facilities such as cut, copy, and paste. You may type, change text, and save, nothing more. If you quit the editor without saving, you will be prompted to save your file.

Because text files in TOS are stored in Unicode format, you may use any human language supported by Unicode, as long as your host’s keyboard and video display permit it. See the Unicode Web site (http://www.unicode.org) for a complete list of Unicode-supported languages.

Write permission is required to edit a file.

Compiling and Running Java Programs for TOS

Selecting a Compiler

TOS does not contain a Java compiler; you will have to make use of one from your host machine. Before doing this you must tell TOS where on your host your compiler is located. Use the regcomp command for this. You may enter the command in any form that would execute successfully from the same host directory from where you launched TOS.

For instance, if the Sun javac compiler is already is your PATH variable, you could enter simply

regcomp javac

If you need to use the Microsoft jvc compiler from a directory not in your path, you might enter something like

regcomp d:\DevStudio\VJ98\bin\jvc.exe

Note: TOS does not handle host filenames with space characters, as Windows does. You may wish to make use of batch files to get around this.

Compiling Java Source Files

Once you have registered a host compiler, you may run it with the javac command. Note that javac does not mean that the host compiler must have that name; as explained in the preceding section, any Java compiler on the host may be used.

The syntax is:

javac filename[.java] [parameters]

The source file must have the extension .java, but this can be omitted from the command.  The remaining parameters will be passed to the host compiler.

The compiler will execute in the directory on your host from which you started the TOS console. You must ensure that the CLASSPATH variable on your host includes the paths of any Java packages you are importing, including the TOS packages.

See the Developer’s Guide for information on writing Java applications for TOS.

Running Java Programs

Use the Run command to run a Java class file. You may run the class either in the foreground, so that its standard input and output map to the console, or in the background, completely separate from the console.

Note, however, that in either case TOS exports the file to the directory on your host from which you ran the TOS console and runs the class from there, using the executable on your host contained in your host’s JAVA environment variable. You must make sure that any classes referenced in your applications are contained in your host’s CLASSPATH environment variable.

Although TOS will export the actual files you will be using, it does not export any supporting classes you have written. These must be exported in advance to a location in your host’s CLASSPATH.

You may not use package names. Java classes to be run from TOS must be part of an anonymous package. Use of a package name will cause a run-time error from your host Java interpreter.

Importing and Exporting Files from TOS to Your Host

Text and Binary Modes

TOS text files are stored using the Unicode convention, which uses two bytes per character. TOS will, however, take care of translating between Unicode and the default text encoding system of your host. On both UNIX and English-language versions of Windows, this is ASCII.

This does mean, however, that when importing or exporting a file you must specify whether it is a binary or text file.

Importing Files

To import a file, use the import command:

import host-source-name tos-destination-name mode [password]

You must have write permission on the TOS file tos-destination, and be able to read the host file host-source. Indicate either "text" or "binary" as the mode parameter. There is no default, so if the mode is omitted you will receive an error message.

Host filenames are relative to the directory from which you ran the TOS console.

Exporting Files

To export a file, use the export command:

export tos-source host-destination mode [password]

You must have read permission on the TOS file tos-destination, and be able to write to the host file host-destination. Indicate either "text" or "binary" as a the mode parameter. There is no default, so if the mode is omitted you will receive an error message.

Host filenames are relative to the directory from which you ran the TOS console.

TOS Script Files

TOS has the ability to place multiple commands in a single text file and execute them as a group. A script file must have the extension .scr.

To run a script file, type

exec script-filename[.scr] [password]

Actually typing the .scr extension is optional. You must have execute permission on the script file in order to run it.

Unlike MS-DOS batch files or UNIX shell scripts, you may only use actual TOS commands in a script file. There are no operators or other statements permitted.

Exiting TOS

Exit TOS with the exit command. You do not need to disconnect before doing this.

TOS Command Summary

Those TOS commands that have counterparts in MS-DOS and the UNIX shells usually may be executed using either name for convenience. Thus, copy and cp have the same effect, as do move and mv, dir and ls etc.

Note, though, that only the UNIX / character may be used as a path separator. The \ character (the MS-DOS path separator) is just another character in TOS.

The TOS console is case-sensitive throughout.

Parameters listed within brackets [] are optional.

chmod

chmod filename permissions password

Change permissions on a file.

filename - Name of file
permissions– New permissions
password– Password. May not be blank.

 

connect

connect host[:port]

Connect to a TOS launcher.

host– Name or IP address of a launcher to connect to.
port – TCP/IP port the launcher is running on at the host.  Defaults to 1099.

 

copy

copy source-file destination-file
cp source-file destination-file

Copies a file.

source-file – File to be copied.
destination-file – New file

 

delete

delete filename
del filename
rm filename

Deletes a file.

filename– Name of file to delete. Write permission is required.

 

dir

dir
ls

Lists the contents of the current directory.

 

disconnect

disconnect

Disconnect from the current TOS launcher.

edit

edit filename password

Invokes the TOS editor and loads the file in question, or creates a new file if none exists.
filename – File to edit. If it exists already, must have read and write permission.
password– Password of file, if necessary.

 

exec

exec script-file[.scr] [password]

Executes a script file.

script-file – Name of script file to execute. Execute permission required.
password – Password, if necessary.

 

exit

exit

Exits TOS.

 

export

export tos-source host-destination mode [password]

Export a TOS file to a file on the host.

tos-source – TOS file to export. Read permission is required on this file.
host-destination – Name of new file on host.
mode – Whether to treat the TOS file as a text or binary file.
password –Password of TOS file, if necessary

 

javac

javac filename[.java] [parameters]

Invokes a host Java compiler.

filename[.java] – Name of Java source file on TOS. File extension is optional.
parameters – Parameters to be sent to the compiler.

 

import

import host-source tos-destination mode [password]

Imports a host file into a TOS file.

host-source – Name of host file to import.
tos-destination – Name of new TOS file. Write permission is required on this file if it already exists.
mode - Whether to treat the TOS file as a text or binary file.
password – Password of TOS file, if necessary

 

move / mv

move source-file destination-file mv source-file destination-file

Creates a copy of a file at another location and deletes the original file.

source-file – File to be copied and deleted.
destination-file – New file.

 

mkdir / md

mkdir dirname
md dirname

Creates a directory.

dirname – Name of new directory.

 

protect

protect filename [oldpassword]

Protects a file and sets a new password.

oldpassword – The former password, if there is one.

 

regcomp

regcomp compiler

Registers a host compiler.

compiler – Name of compiler on host to be invoked with the javac command.

 

rmdir / rd

rmdir dirname
rd dirname

Removes a directory. All files in the directory are also deleted without warning.

dirname – Name of directory to remove.

 

run

run class[.class] [&]

Runs a Java class file on the host.

class – Name of class to execute. Typing the extension .class is optional. Package names are qualified with the / operator instead of a period. Execute permission required.
& – If absent, indicates foreground operation, so the running class’ standard input and output are connected to the TOS console, which halts until the class finishes. If present, class runs asynchronously in the background.
